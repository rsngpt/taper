The problem

Agent-integrated apps hit three failure modes that hurt most on the budget devices (4–6GB RAM) most users actually own.

💥 DOM parsing OOMs

body.string() → org.json materialises the whole payload as a tree of maps, boxed numbers, and string headers — a multiple of the raw size, inside a ~192MB default app heap.

🔁 Blind retries

A malformed tool call or policy rejection can never succeed on retry — but naive loops retry it anyway, burning tokens, battery, and money. A radio blip, meanwhile, deserves the retry it often doesn't get.

📶 Reconnect floods

Going offline mid-conversation queues updates; coming back online fires them one request at a time — redundant network chatter exactly when the connection is weakest.

Three components, one dependency

Kotlin, minSdk 24, no cloud dependency. Each piece works alone; together they cover the ingest → classify → sync loop.

1 · Streaming parser

Declare fields of interest; everything else is skipped token-by-token and never allocated. Cost scales with what you extract, not document size.

val result = TaperParser().parse(
  inputStream,
  fieldsOfInterest = setOf(
    "model",
    "messages[].content",
    "usage.total_tokens",
  ),
)
val tokens =
  result.firstLong("usage.total_tokens")

2 · Exception classifier

Rules-based triage: SEMANTIC failures halt (they can never succeed on retry); TRANSIENT ones queue. 429 is transient despite being 4xx; 501 is semantic despite being 5xx.

when (ExceptionClassifier().classify(
  AgentFailure(httpStatus = 429)
)) {
  TRANSIENT -> queue.enqueue(convId, payload)
  SEMANTIC  -> surfaceToUser()
}

3 · Offline sync queue

Room-backed, survives process death, and on reconnect coalesces every pending update into one batch per conversation instead of a request flood.

val queue = SyncQueue.create(context)
queue.enqueue("conv-42", updateJson)
ConnectivityDrainTrigger(
  context, queue, syncer, scope
).start()

How it fits together

Failures loop through the classifier; only transient ones re-enter the durable queue. The ParserEngine interface is the seam where a native/ashmem engine plugs in later without breaking the public API.

Honest scope

What v1 is — and deliberately isn't.

  • Built & tested: all three components, 64 unit tests (incl. a 32MB-heap child-JVM proof that DOM parsing dies where Taper streams, and process-death recovery against the real database), instrumented benchmark with per-test process isolation, CI on every push.
  • Not built, by design: FlatBuffers/ashmem/JNI engines (interface seam only), ML failure classification (rules are auditable and the signals are documented contracts; a logistic-regression tie-breaker is the scoped v2), cloud gateway / context compression.
  • Every number on this page was measured by the harness in this repo — on one emulator profile. Different payload shapes and devices will differ; the harness is one script away.
